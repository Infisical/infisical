Bypass Approvals is a per-policy escape hatch for emergency PAM access. When enabled on a policy, designated users (called **bypassers**) can approve their own pending access requests without waiting for the configured approvers — provided they supply a written reason. The bypass is fully audited and notifies all approvers.

Bypass Approvals is supported only for PAM access policies.

## When to Use Bypass Approvals

Enable Bypass Approvals when:

- **An incident requires immediate access**: An on-call engineer needs to act on a production issue and waiting for an approver would extend the outage.
- **You need a documented escape hatch**: Compliance requires that emergency access is recorded with a reason and notified to approvers.

## Configuring Bypass Approvals

Project admins enable Bypass Approvals when creating or editing a PAM access policy.

1. Navigate to **PAM Project > Approvals > Policies** and create or edit a policy.
2. Enable the **Bypass Approvals** toggle.
3. Add **User Bypassers** and/or **Group Bypassers** — the users and groups allowed to bypass approvals on requests matching this policy.

If Bypass Approvals is enabled but no bypassers are selected, **anyone in the project** can bypass this policy. The UI surfaces a warning when you save in this state.

## Approving via Bypass

When a bypasser opens their own pending request, the review section includes an additional checkbox: *"Approve immediately without approver review (bypass policy)"*. Checking it reveals a required reason field (minimum 10 characters). Submitting:

- Marks the request as Approved and issues an access grant tagged as bypassed.
- Records a `pam-access-policy-bypassed` audit event with the actor, target, duration, and reason.
- Sends an in-app notification and email to every approver configured on the request.

## Who Can Bypass

A user can bypass a request only if **all** of the following are true at the moment they submit:

- They are signed in as a user (machine identities cannot bypass).
- They are the requester of the pending request.
- The policy still has Bypass Approvals enabled.
- They are listed as a bypasser on the policy (or the bypassers list is empty).

These checks run at submission time, not at request creation. If an admin changes the policy between request creation and the bypass attempt, the new policy state applies.

## What Gets Audited

Each bypass approval emits a `pam-access-policy-bypassed` audit event with the policy ID, request ID, grant ID, grantee, target resource and account, access duration, the bypass reason, and the number of approvers notified. Find these in **Audit Logs** filtered by event type *PAM Access Policy Bypassed*.

The standard *Approve Approval Request* event is not emitted for a bypass approval — the bypass event replaces it.
