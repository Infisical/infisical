If you're on the legacy project-based Certificate Manager, this guide explains what changed and how to migrate to the new Application-centric model.

<Warning>
  This guide is for existing customers on the legacy model. If you're new to Infisical Certificate Manager, start with [Getting Started](/documentation/platform/pki/getting-started).
</Warning>

## Backward Compatibility

Your existing certificates, profiles, alerts, syncs, and approval policies continue to work. Certificates will keep renewing, alerts will keep firing, and syncs will keep running.

**What you can still do at the platform level:**
- Create new Certificate Authorities, Policies, and Profiles
- Manage existing resources

**What must now be done inside Applications:**
- Configure enrollment methods (API, ACME, EST, SCEP)
- Create alerts
- Create certificate syncs
- Define approval policies

Legacy profiles that had enrollment methods configured will continue to work, but you cannot modify their enrollment configuration. New issuance must go through [Applications](/documentation/platform/pki/applications/overview).

This is the new model going forward. We encourage you to migrate your workflows to Applications to take advantage of the improved access control and organizational structure.

## What Changed

The new Certificate Manager removes the "project" concept and introduces a clearer separation between platform-level infrastructure (CAs, Policies, Profiles) and day-to-day operations (Applications).

### Key Changes

| Before | After |
|--------|-------|
| Certificate Management Project | Certificate Manager |
| Certificates issued at project level | Certificates issued through Applications |
| Profiles had enrollment methods baked in | Enrollment methods configured per Application |
| Alerts filtered by profile/properties | Alerts configured per Application |
| Syncs configured at project level | Syncs configured per Application |
| Approval policies defined separately | Approval policies defined inside Applications |
| Access control at project level | Two-level access: Platform + Application |

### New Hierarchy

```
Organization
└── Certificate Manager
    ├── Platform-Level (Admin manages)
    │   ├── Certificate Authorities
    │   ├── Certificate Policies
    │   ├── Certificate Profiles
    │   └── Requests (centralized approvals)
    │
    └── Applications (Teams operate)
        ├── Enrollment Methods
        ├── Certificates
        ├── Alerts
        ├── Syncs
        └── Approval Policies
```

### New Access Control Model

**Platform Level:**

| Role | What they see | What they can do |
|------|---------------|------------------|
| **Platform Admin** | Everything | Full control of CAs, Policies, Profiles, Applications |
| **Platform Member** | Applications they're assigned to | Operate within assigned Applications only |

**Application Level:**

| Role | What they can do |
|------|------------------|
| **Admin** | Configure enrollment methods, syncs, alerts, approvals. Manage members. |
| **Operator** | Issue, renew, revoke certificates |
| **Auditor** | View certificates and configuration (read-only) |

## Migration Steps

<Steps>
  <Step title="Review your current setup">
    Document your existing:
    - Certificate profiles and their enrollment methods
    - Alert configurations
    - Certificate syncs
    - Approval policies
    - Team members and their access levels
  </Step>
  <Step title="Create Applications">
    As a Platform Admin, create an Application for each logical grouping of certificates (e.g., per service, per team):
    
    1. Go to **Certificate Manager → Applications**
    2. Click **Create Application**
    3. Name it after the service or team (e.g., `payments-api`, `platform-team`)
  </Step>
  <Step title="Configure enrollment methods">
    For each Application:
    
    1. Go to **Enrollment Methods** within the Application
    2. Add the appropriate method(s) — API, ACME, EST, or SCEP
    3. Select the certificate profile that defines certificate attributes
    
    <Note>
      Profiles no longer have enrollment methods baked in. You configure enrollment at the Application level, allowing one profile to be used across multiple Applications with different enrollment methods.
    </Note>
  </Step>
  <Step title="Recreate alerts">
    For each Application:
    
    1. Go to **Alerting** within the Application
    2. Create alerts for expiration, issuance, renewal, and revocation
    
    <Note>
      Your existing alerts will continue working, but you cannot modify them. New alerts must be created inside Applications and apply to all certificates in that Application.
    </Note>
  </Step>
  <Step title="Recreate syncs">
    For each Application:
    
    1. Go to **Certificate Syncs** within the Application
    2. Configure syncs to push certificates to your destinations
    
    <Note>
      Your existing syncs will continue working, but you cannot modify them. New syncs must be created inside Applications.
    </Note>
  </Step>
  <Step title="Recreate approval policies">
    For each Application:
    
    1. When attaching a profile, configure approval requirements
    2. Approval policies are now scoped to the Application + Profile combination
    
    Pending approval requests appear in the **Approval Requests** page in the sidebar.
  </Step>
  <Step title="Assign team members">
    1. Grant **Platform Member** role to team members who need Application access
    2. Assign them to specific Applications with **Admin**, **Operator**, or **Auditor** roles
  </Step>
  <Step title="Update issuance clients">
    Enrollment URLs have changed from profile-based to Application + Profile based. Update your:
    
    - **ACME clients** (Certbot, cert-manager, etc.) — Directory URLs now include both Application ID and Profile ID
    - **Infisical Agent** — Update configuration to reference `application-name` and `profile-name`
    - **EST/SCEP clients** — Endpoint URLs now include Application and Profile path segments
    - **Terraform** — Update any `infisical_pki_*` resources to use the new Application-based configuration
    - **API integrations** — Review endpoints for path changes and update to use the new Application context
  </Step>
</Steps>

## FAQ

<AccordionGroup>
  <Accordion title="Will my existing certificates keep working?">
    Yes. All existing certificates, renewals, alerts, and syncs continue to work exactly as before. Nothing will break.
  </Accordion>
  <Accordion title="Can I still modify my existing profiles/alerts/syncs?">
    No. Legacy resources are frozen — they continue to work, but you cannot reconfigure them. New resources must be created inside Applications.
  </Accordion>
  <Accordion title="Do I need to re-issue certificates?">
    No. Existing certificates are unaffected. The migration is about how you organize and manage certificates going forward.
  </Accordion>
  <Accordion title="What if I don't migrate?">
    Your existing setup will continue working, but you won't be able to create new profiles, alerts, or syncs at the platform level. All new configuration must happen inside Applications.
  </Accordion>
  <Accordion title="Can I still use certificate profiles?">
    Yes. Profiles still exist and define certificate attributes (CA, validity, constraints). The change is that enrollment methods are now configured per Application, not per profile.
  </Accordion>
</AccordionGroup>
