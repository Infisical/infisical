In Infisical, code signing approval workflows add a human review step before signing operations can be performed, helping organizations enforce separation of duties and maintain compliance for software supply chain security.

## When to Use Approval Workflows

Approval workflows are recommended when:

- **Separation of duties is required**: Your organization requires different people to request and authorize code signing.
- **Release signing needs oversight**: Signing production releases, firmware, or customer-facing artifacts requires additional review.
- **Compliance mandates review**: Regulatory frameworks or internal policies require documented approval before signing.
- **Preventing unauthorized signing**: You want to ensure artifacts are only signed after proper validation.

## Approval Modes

Infisical supports three approval modes that control how signing grants are issued:

### Manual

Each approved request grants exactly **one signing operation**. This is the strictest mode. Every artifact signature requires a separate approval.

This mode is recommended for high-security environments where each signing operation must be individually authorized.

### Time Window

Each approved request grants signing access for a **defined time period**. You can sign unlimited times within the window.

This mode is recommended for release cycles where a team needs to sign multiple artifacts during a release window (e.g., a 4-hour release deployment).

<Note>
  - The policy defines a `maxWindowDuration` that constrains how long a window can be (from 1 minute to 30 days).
  - The requester specifies the desired `windowStart` and `windowEnd` within that constraint.
</Note>

### N-Signings

Each approved request grants a **fixed number of signing operations**. Once the quota is exhausted, the grant expires automatically.

This mode is recommended for CI pipelines that need to sign a known number of artifacts per build (e.g., 10 JARs per release).

<Note>
  The policy defines a `maxSignings` that caps the number of signings a requester can ask for.
</Note>

## Approval Policies

An approval policy defines the workflow that must be completed before signing access is granted for a specific signer. Every [signer](/documentation/platform/pki/code-signing/signers) must be linked to an approval policy.

Key features of approval policies include:

- **Approval mode selection**: Choose between Manual, Time Window, or N-Signings mode.
- **Multi-step workflows**: Configure sequential approval steps where each step must be completed before the next begins.
- **Flexible approvers**: Assign individual users or groups as eligible approvers for each step.
- **Required approval count**: Specify how many approvals are needed per step (e.g., require 2 out of 5 eligible approvers).
- **Request expiration**: Set a maximum time-to-live (TTL) for pending requests.

### Guide to Creating an Approval Policy

To create an approval policy, head to your Certificate Management Project > **Code Signing > Approvals > Policies** and press **Create Policy**.

<Steps>
  <Step title="Configuring the policy">
    Configure the basic policy settings:

    - **Policy Name**: A descriptive name for the policy such as `release-signing-approval`.
    - **Approval Mode**: The mode that fits your workflow:
      - **Manual**: One approval = one signature.
      - **Time Window**: Approval grants access for a time period. Configure the **Max. Window Duration** (e.g., `4h`, `1d`).
      - **N-Signings**: Approval grants a fixed quota. Configure the **Max. Signings** count.
    - **Max. Request TTL**: The maximum time a request can remain pending before it expires (optional).
    - **Bypass approval for machine identities**: When enabled, machine identities can sign without requiring approval.

    ![Create approval policy - configuration](/images/platform/pki/code-signing/approval-policy-create.png)
  </Step>
  <Step title="Configuring the approval sequence">
    Configure the approval steps. Each step defines who can approve and how many approvals are required:

    - **Step Name**: An optional name for the step such as `Security Team Review`.
    - **Approvers**: The individual users or groups who are eligible to approve this step.
    - **Required Approvals**: The number of approvals needed to complete this step.

    You can add multiple steps to create a sequential approval workflow. For example:
    1. **Team Lead Review**: Requires 1 approval from the team leads group
    2. **Security Review**: Requires 2 approvals from the security team

    Each step must be completed in order before the signing grant is issued.

    ![Create approval policy - approval sequence](/images/platform/pki/code-signing/approval-policy-steps.png)
  </Step>
  <Step title="Reviewing and creating the policy">
    Review your policy configuration and press **Create** to save the policy.

    ![Create approval policy - review](/images/platform/pki/code-signing/approval-policy-review.png)
  </Step>
</Steps>

## Requesting Signing Access

When a user or machine identity needs to sign artifacts, they must first request signing access through the approval workflow.

### Guide to Requesting Access

Head to your Certificate Management Project > **Code Signing > Approvals > Requests** and press **Request Signing Access**.

![Approval requests page](/images/platform/pki/code-signing/approval-requests.png)

<Steps>
  <Step title="Selecting a signer and configuring parameters">
    Select the signer you want to use. The form will automatically load the approval mode associated with that signer's policy.

    Depending on the approval mode, provide the required parameters:

    - **Manual**: No additional parameters needed.
    - **Time Window**: Specify the **Window Start** and **Window End** for when you need signing access.
    - **N-Signings**: Specify the **Number of Signings** you need.
    - **Justification**: Optionally provide a reason for the request (e.g., "Release v2.4.0 signing").

    ![Request signing access](/images/platform/pki/code-signing/request-access-params.png)
  </Step>
  <Step title="Submitting the request">
    Press **Submit** to create the request. Eligible approvers will be notified.
  </Step>
</Steps>

## Approval Requests

When a signing access request is submitted, an approval request is created. Approvers can then review and approve or reject the request.

### Viewing Requests

Head to **Code Signing > Approvals > Requests** to view all approval requests. You can filter requests by status:

- **Open**: Requests currently pending approval
- **Approved**: Requests that have been approved and grants issued
- **Rejected**: Requests that were rejected by an approver
- **Cancelled**: Requests cancelled by the requester
- **Expired**: Requests that exceeded their maximum TTL

### Approving a Request

<Steps>
  <Step title="Opening the request">
    Press on a pending request to view its details.

    ![Approval request detail](/images/platform/pki/code-signing/approval-request-detail.png)
  </Step>
  <Step title="Reviewing the request details">
    Review the signing access request information including:
    - Requester name
    - Signer name and certificate
    - Approval mode and parameters (time window, signing count, etc.)
    - Justification
  </Step>
  <Step title="Approve or add comments">
    If you are an eligible approver for the current step, press **Approve** to approve the request.
  </Step>
</Steps>

Once all required approvals for all steps are obtained, a signing grant is automatically issued and the requester can begin signing.

### Rejecting a Request

<Steps>
  <Step title="Opening the request">
    Press on a pending request to view its details.
  </Step>
  <Step title="Rejecting with reason">
    If you are an eligible approver for the current step, press **Reject** to reject the request. Optionally add a comment explaining the rejection.
  </Step>
</Steps>

When a request is rejected, no signing grant is issued.

## Grants

A grant is the authorization that allows a user or machine identity to perform signing operations with a specific signer. Grants are automatically created when an approval request is fully approved.

### Viewing Grants

Head to **Code Signing > Approvals > Grants** to view all grants. You can filter by status:

- **Active**: Grants that are currently valid and can be used for signing.
- **Expired**: Grants that have expired (time window ended, signing quota exhausted, or TTL reached).
- **Revoked**: Grants that were manually revoked by an administrator.

![Grants page](/images/platform/pki/code-signing/grants.png)

### Revoking a Grant

Administrators can revoke an active grant at any time. To revoke a grant, press the **Revoke** action on the grant row and optionally provide a reason.

Once revoked, the grant can no longer be used for signing operations. The user must request a new grant through the approval workflow.

## FAQ

<AccordionGroup>
  <Accordion title="I approved a request but the grant wasn't issued">
    If the approval policy has multiple steps, your approval may have completed only one step. The grant is issued only after all approval steps are completed. Check the request details to see which step is currently pending.
  </Accordion>
  <Accordion title="My grant expired before I finished signing">
    Depending on the approval mode:
    - **Time Window**: Your window end time has passed. Request a new grant with a longer window.
    - **N-Signings**: You used all your allotted signings. Request a new grant with a higher count.
    - **Manual**: Manual grants are single-use. Request a new grant for each signing operation.
  </Accordion>
  <Accordion title="Can I extend an active grant?">
    Grants cannot be extended. You need to create a new approval request for additional signing access.
  </Accordion>
  <Accordion title="What happens to in-flight signing operations when a grant is revoked?">
    If a signing operation is already being processed when the grant is revoked, the operation will complete. Subsequent signing attempts with the revoked grant will be denied.
  </Accordion>
</AccordionGroup>
