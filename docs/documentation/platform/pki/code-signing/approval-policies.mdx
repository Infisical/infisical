In Infisical, code signing approval workflows add a human review step before signing operations can be performed, helping organizations enforce separation of duties and maintain compliance for software supply chain security.

## When to Use Approval Workflows

Approval workflows are recommended when:

- **Separation of duties is required**: Your organization requires different people to request and authorize code signing.
- **Release signing needs oversight**: Signing production releases, firmware, or customer-facing artifacts requires additional review.
- **Compliance mandates review**: Regulatory frameworks or internal policies require documented approval before signing.
- **Preventing unauthorized signing**: You want to ensure artifacts are only signed after proper validation.

## Approval Modes

Infisical supports three approval modes that control how signing grants are issued:

### Manual

Each approved request grants exactly **one signing operation**. This is the strictest mode. Every artifact signature requires a separate approval.

This mode is recommended for high-security environments where each signing operation must be individually authorized.

### Time Window

Each approved request grants signing access for a **defined time period**. You can sign unlimited times within the window.

This mode is recommended for release cycles where a team needs to sign multiple artifacts during a release window (e.g., a 4-hour release deployment).

<Note>
  - The policy defines a `maxWindowDuration` that constrains how long a window can be (from 1 minute to 30 days).
  - The requester specifies the desired `windowStart` and `windowEnd` within that constraint.
</Note>

### N-Signings

Each approved request grants a **fixed number of signing operations**. Once the quota is exhausted, the grant expires automatically.

This mode is recommended for CI pipelines that need to sign a known number of artifacts per build (e.g., 10 JARs per release).

<Note>
  The policy defines a `maxSignings` that caps the number of signings a requester can ask for.
</Note>

## Creating an Approval Policy

An approval policy defines the workflow that must be completed before signing access is granted for a specific signer.

Key features of approval policies include:

- **Approval mode selection**: Choose between Manual, Time Window, or N-Signings mode.
- **Multi-step workflows**: Configure sequential approval steps where each step must be completed before the next begins.
- **Flexible approvers**: Assign individual users or groups as eligible approvers for each step.
- **Required approval count**: Specify how many approvals are needed per step (e.g., require 2 out of 5 eligible approvers).
- **Request expiration**: Set a maximum time-to-live (TTL) for pending requests.

To create an approval policy, head to your Certificate Management Project > **Code Signing > Approvals > Policies** and press **Create Policy**.

<Steps>
  <Step title="Configuring the policy">
    Configure the basic policy settings:

    - **Policy Name**: A descriptive name for the policy such as `release-signing-approval`.
    - **Approval Mode**: The mode that fits your workflow:
      - **Manual**: One approval = one signature.
      - **Time Window**: Approval grants access for a time period. Configure the **Max. Window Duration** (e.g., `4h`, `1d`).
      - **N-Signings**: Approval grants a fixed quota. Configure the **Max. Signings** count.
    - **Max. Request TTL**: The maximum time a request can remain pending before it expires (optional).
    - **Bypass approval for machine identities**: When enabled, machine identities can sign without requiring approval.

    ![Create approval policy - configuration](/images/platform/pki/code-signing/approval-policy-create.png)
  </Step>
  <Step title="Configuring the approval sequence">
    Configure the approval steps. Each step defines who can approve and how many approvals are required:

    - **Step Name**: An optional name for the step such as `Security Team Review`.
    - **Approvers**: The individual users or groups who are eligible to approve this step.
    - **Required Approvals**: The number of approvals needed to complete this step.

    You can add multiple steps to create a sequential approval workflow. For example:
    1. **Team Lead Review**: Requires 1 approval from the team leads group
    2. **Security Review**: Requires 2 approvals from the security team

    Each step must be completed in order before the signing grant is issued.

    ![Create approval policy - approval sequence](/images/platform/pki/code-signing/approval-policy-steps.png)
  </Step>
  <Step title="Reviewing and creating the policy">
    Review your policy configuration and press **Create** to save the policy.

    ![Create approval policy - review](/images/platform/pki/code-signing/approval-policy-review.png)
  </Step>
</Steps>
